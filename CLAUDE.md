# CLAUDE.md

Гайд для ИИ-ассистентов (Claude Code и др.), работающих в этом репозитории.
Описывает **обвязку харнеса**, архитектуру проекта, конвенции и правила.
Прочитай этот файл первым, прежде чем менять код.

---

## 0. TL;DR (самое важное)

- **Проект:** `STARWARS — AR Space Fighter` — AR-шутер про космический истребитель.
  Стек: **WebXR + Three.js**, AR-lite (камера + гироскоп), без билд-степа.
- **Рабочая ветка: `main`.** НЕ создавай других веток. Коммить и пушь в `main`.
- **Билд не нужен:** это статические ES-модули; Three.js грузится с CDN через import map.
  Чтобы запустить — поднимай любой статический сервер из корня.
- **AR требует HTTPS** (secure context) для камеры и DeviceOrientation. Локально —
  `localhost` ок для камеры; для теста с телефона нужен HTTPS (туннель или Netlify).
- **Деплой:** статика на Netlify, конфиг в `netlify.toml` (уже настроен).
- Все тюнинг-параметры геймплея/графики/аудио живут в `src/config.js` — меняй там, а не
  «магическими числами» по коду.

## 1. Обвязка харнеса (Harness Wiring)

Этот раздел описывает, как окружение, в котором запускается ассистент, связано с
проектом, и какие из этого следуют правила.

### 1.1 Среда исполнения

- Ассистент работает в **эфемерном облачном контейнере** (Claude Code on the web).
  Репозиторий клонируется заново при старте сессии; контейнер уничтожается по
  завершении. **Всё ценное должно быть закоммичено и запушено**, иначе потеряется.
- Платформа: Linux. Рабочая директория: корень репозитория.
- Сеть контейнера ограничена сетевой политикой окружения. **Важно:** Three.js с CDN
  грузится в браузере *игрока*, а не в контейнере, поэтому CDN-зависимость не упирается в
  сетевую политику сборки.

### 1.2 Контроль версий и ветки

- **Единственная рабочая ветка — `main`** (так распорядился владелец репозитория).
- НЕ создавай feature-веток, НЕ переключай ветки без явной просьбы.
- Коммить осмысленными порциями, понятными сообщениями (см. §6).
- Пуш: `git push -u origin main`. При сетевых сбоях — ретраи с экспоненциальной
  паузой (2s, 4s, 8s, 16s), до 4 попыток.
- **PR не создавать**, если пользователь явно не попросил.

### 1.3 GitHub-обвязка

- Прямого доступа к `gh` CLI нет. Все операции с GitHub — через инструменты
  `mcp__github__*` (просмотр/создание PR, комментарии, CI и т.п.).
- Текущая область видимости репозиториев ограничена настройками сессии.

### 1.4 Деплой-обвязка (Netlify)

- Деплой — **статический**, без команды сборки (`netlify.toml: command = ""`,
  `publish = "."`).
- Netlify раздаёт по HTTPS → AR (камера/сенсоры) работает на проде из коробки.
- В `netlify.toml` выставлен `Permissions-Policy` для `camera`, `gyroscope`,
  `accelerometer`, `magnetometer` — не удаляй, иначе AR-ввод отвалится.

### 1.5 Что НЕ делать

- Не добавляй тяжёлый бандлер/тулчейн без необходимости — ценность проекта в
  «zero-build, открыл и работает».
- Графика — **процедурная** (партиклы/шейдеры/примитивы), бинарные модели не коммитим.
- Звук — **реальные royalty-free ассеты** (Kenney CC0 SFX + FreePD public-domain музыка),
  лежат в `assets/audio/`, играются через Howler.js (вендорнут в `vendor/`). Это
  обновлённое решение: процедурный синтез звука забракован как «не AAA». Любой новый
  аудиофайл — только с проверяемой лицензией и записью в `assets/audio/CREDITS.md`.
- Не хардкодь секреты. Их тут и не должно быть (чистый фронтенд).

## 2. Структура репозитория

```

│   ├── core/

│   │   └── Input.js        # DeviceOrientation (гироскоп) + тач/свайп + кнопки
│   ├── ar/
│   │   └── ARCamera.js     # getUserMedia фон с камеры + наведение по ориентации

│   │   ├── Starfield.js    # Процедурное звёздное небо (Points)
│   │   ├── Nebula.js       # Туманность на шейдере (аддитивная, поверх AR-фона)
│   │   └── Postprocessing.js # EffectComposer + UnrealBloom (HDR-свечение)
│   ├── entities/
│   │   ├── Player.js       # Состояние игрока: щит/корпус/буст, пушки
│   │   ├── EnemyFighter.js # Враг: геометрия, ХП, тип, ссылка на ИИ
│   │   ├── Projectile.js   # Снаряд-трассер (пулится)
│   │   └── Explosion.js    # Партиклы взрыва (пулится)
│   ├── systems/
│   │   ├── Spawner.js      # Волны, состав, рост сложности, боссы
│   │   ├── EnemyAI.js      # Стейт-машина поведения врага
│   │   ├── CombatSystem.js # Коллизии снаряд↔корабль, урон, события убийств
│   │   └── ScoreSystem.js  # Очки, комбо, рекорд (localStorage)
│   ├── audio/
│   │   └── AudioEngine.js  # Web Audio: синтез SFX + адаптивная музыка, 3D-паннинг
│   └── ui/
│       ├── HUD.js          # Прицел, щит/корпус, счёт, комбо, волна, радар
│       └── Screens.js      # Брифинг / пауза / game over / victory

├── tools/                  # Вспомогательные скрипты (проверка синтаксиса и т.п.)

├── package.json            # Метаданные + dev-скрипты (статик-сервер)
```

## 3. Архитектура и поток данных

```
index.html → src/main.js → core/Game.js
                              │
        ┌─────────────────────┼───────────────────────────┐
        ▼                     ▼                            ▼
   ar/ARCamera          world/* (сцена,            core/Input (гироскоп/тач)
   (видео-фон +          звёзды, туманность,              │
    ориентация)          постобработка)                   ▼
        │                     ▲                     core/StateMachine
        ▼                     │                            │
   THREE.Camera ◄─────────────┘                            ▼
                                                  systems/* каждый кадр:
                                                  Spawner → EnemyAI → Combat → Score
                                                            │
                                                            ▼
                                                  entities/* (Player, Enemy,
                                                  Projectile, Explosion) — пул
                                                            │
                                                            ▼
                                                  ui/HUD + ui/Screens + audio/AudioEngine
```

**Игровой цикл** (в `Game.js`) на `requestAnimationFrame`:
`update(dt)` → системы по порядку → `render()` (через EffectComposer).

**Принципы:**
- **Один источник правды для тюнинга** — `config.js`.
- **Пулинг** для снарядов и взрывов (без аллокаций в горячем цикле).
- **Системы не лезут в DOM** напрямую (кроме UI-слоя); связь — через события/коллбеки.
- **Состояния игры** маршрутизируются через `StateMachine`; ввод и апдейт зависят от
  текущего состояния.

## 4. Локальный запуск и проверка

```bash
# Поднять статику из корня (любой вариант):
npx serve . -l 8080
python3 -m http.server 8080
```
Открыть `http://localhost:8080`.

- Камера/гироскоп: на десктопе гироскопа нет → работает фолбэк-свайп; камера может
  отсутствовать → AR-фон деградирует в чистый космос. Это ожидаемо.
- Проверка синтаксиса модулей без браузера: `npm run check` (см. `tools/`).
- Полноценный тест AR — только с телефона по HTTPS (Netlify-превью или туннель).

## 5. Конвенции кода

### 5.0 Философия кода — «техника Карпаты» (обязательно)

Код пишем в духе Андрея Карпаты (nanoGPT / micrograd): **минимально, от первооснов,
читаемо как учебник**. Конкретно:

- **Маленькие сфокусированные модули.** Один файл — одна ясная ответственность. Если
  файл разросся и делает «много всего» — раздели.
- **От первооснов, без лишних абстракций.** Не вводи фабрики/менеджеры/слои ради
  «архитектурности». Пиши прямую, понятную механику. Абстракция появляется только когда
  её требует *реальное* повторение, а не гипотетическое.
- **Ноль лишних зависимостей.** Совпадает с zero-build: только Three.js (CDN) и
  браузерные API. Не тащи библиотеку ради того, что пишется в 20 строк.
- **Читаемость важнее «умности».** Явные имена, прямой поток управления, никакой
  криптовой однострочной магии. Код должен читаться сверху вниз.
- **Комментируй *почему*, а не *что*.** Объясняй неочевидные решения, числа, физику,
  трюки рендера/аудио. Очевидное не комментируй.
- **Инкрементально и проверяемо.** Небольшие рабочие куски; каждый прогоняй через
  `npm run check` (синтаксис) и, по возможности, в браузере.
- **Горячий путь — простой и быстрый.** В `update`/`render` без аллокаций, без хитрых
  замыканий на кадр; пул объектов вместо мусора для GC.

### 5.1 Технические правила

- **ES-модули**, `type="module"`. Импорт Three.js: `import * as THREE from 'three'`
  (резолвится через import map в `index.html`). Аддоны: `three/addons/...`.
- Без TypeScript (намеренно zero-build). Документируй публичные методы JSDoc-комментами.
- Имена: `PascalCase` для классов/файлов-классов, `camelCase` для функций/переменных,
  `UPPER_SNAKE` для глобальных констант.
- Никаких «магических чисел» в логике — выноси в `config.js`.
- Горячий цикл (`update`/`render`) — без аллокаций и без `console.log`.
- Комментарии — по делу, на языке окружающего кода (в проекте — русский/英 смешанно,
  держись стиля файла).

## 6. Гит-процесс (для ассистента)

- Ветка: **только `main`**.
- Коммить логическими порциями. Формат сообщения:
  ```
  <область>: <что сделано в повелительном наклонении>

  <опционально: почему / детали>
  ```
  Примеры: `world: add shader nebula with additive blending`,
  `systems: implement wave spawner and boss cadence`.
- В конце сообщений коммитов добавляй трейлеры, предписанные харнесом
  (Co-Authored-By / Claude-Session), если они заданы окружением.
- Пуш: `git push -u origin main` (+ ретраи при сетевых ошибках).
- PR — только по явному запросу пользователя.
